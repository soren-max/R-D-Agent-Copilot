# 接口异常排查流程

接口异常排查应先收集事实，再做判断。推荐顺序是：日志 -> 配置 -> 代码变更 -> 历史案例。这样可以先确认请求实际发生了什么，再判断是否由环境差异、近期发布或已知问题引起，避免直接猜测原因。

接口 500 排查流程：先确认接口路径、请求时间、trace_id 和影响范围；再查询 order-service 的 ERROR/WARN 日志，查看是否出现 status=500、异常栈、dependency 或业务错误码。如果日志显示下游失败，需要继续查看下游服务日志；如果日志显示本地异常，则重点检查最近代码变更和输入数据。

接口超时排查流程：先看请求耗时、timeout 配置和下游调用耗时。若日志中出现 payment-service timeout，应确认 payment.timeout 是否过短、下游是否有慢请求、重试是否导致请求堆积。还要检查连接池、线程池和限流日志，判断是单点慢还是系统性拥塞。

配置不一致排查流程：对比 dev、test、prod 中与问题相关的配置，例如 payment.timeout、order.retry_count、feature.enable_new_payment_flow。确认配置版本、发布时间和服务实例是否一致。对于功能开关类配置，需要确认是否有灰度、是否只影响部分流量。

最近代码变更排查流程：查看问题发生前后的提交记录，重点关注异常处理、支付流程、重试策略和配置读取逻辑。高风险提交不一定就是根因，但应优先回查变更文件、测试覆盖和发布批次。如果日志和配置都无法解释现象，再查历史案例和类似事故记录。
