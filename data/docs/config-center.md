# 配置中心说明

配置中心用于统一管理服务运行时需要的配置项，例如超时时间、重试次数、功能开关和下游服务地址。它的价值是把配置从代码中拆出来，让研发和运维可以在不同环境中使用不同参数，并在排障时快速确认线上行为是否符合预期。

常见环境包括 dev、test 和 prod。dev 通常用于本地或联调，配置会更宽松，便于观察和调试；test 用于集成验证，配置应尽量接近生产；prod 是真实生产环境，配置变更必须更谨慎。排查问题时，需要确认用户请求实际命中的环境，避免拿 dev 配置解释 prod 现象。

配置变更不生效通常有几类原因：服务没有拉取到最新配置、配置 key 写错、环境选择错误、缓存未刷新、热更新监听失败，或服务启动时只读取一次配置。还有一种常见情况是配置中心已修改，但实例仍使用旧版本配置，需要结合发布时间和实例重启记录判断。

配置热更新可以减少重启成本，但不代表所有配置都适合动态修改。连接池大小、线程池参数、下游地址等配置可能需要额外初始化逻辑。变更前应确认服务是否支持热更新、是否有灰度机制、是否能回滚，并观察日志中是否出现配置刷新成功的信息。

排查建议：先确认环境和配置版本，再对比 dev、test、prod 的关键差异；重点检查 timeout、retry、feature flag、限流和下游服务配置；最后结合日志中的 trace_id、错误时间点和服务实例信息，判断配置是否与异常现象一致。
